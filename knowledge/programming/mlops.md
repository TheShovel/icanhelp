# Machine Learning Operations (MLOps)

## ML Pipeline Stages
- **Data engineering**: collect (sources: databases, APIs, streams, files, logs, sensors). Validate (schema, data quality constraints — validate at ingestion: null checks, range checks, uniqueness, completeness. Great Expectations, Pandera, Soda). Version (store data snapshots — DVC, Quilt, LakeFS — enable reproducibility, rollback, comparison). Transform (ETL/ELT — dbt, Spark, Python scripts). Feature store (Feast, Tecton — centralized feature registry, serve features for training + inference consistently. Avoid training-serving skew). Lineage tracking for data origin
- **Training**: experiment tracking (MLflow, Weights & Biases, Neptune.ai, Comet, ClearML). Parameters + metrics + artifacts + code version tracked for every run. Hyperparameter tuning (grid search, random search, Bayesian optimization, Hyperband, Optuna, Ray Tune)
- **Evaluation**: holdout test set (don't touch until final evaluation). Cross-validation (k-fold, stratified for classification). Metrics: accuracy, precision, recall, F1, AUC-ROC, log loss, RMSE, MAE, R², business metrics (cost per false positive/negative). Confusion matrix. Fairness evaluation (equalized odds, demographic parity). Model interpretation (SHAP, LIME, permutation importance, partial dependence plots)
- **Deployment**: batch inference (scheduled, process data in bulk). Real-time API (REST/gRPC endpoint. Flask/FastAPI/streaming). Edge (on-device model — TensorFlow Lite, ONNX, CoreML, PyTorch Mobile). Embedded (microcontrollers — TensorFlow Lite Micro). Shadow deployment (run alongside existing model, compare outputs). Canary (roll out to small user %, monitor). Blue-green (switch traffic between two model versions)
- **Monitoring**: data drift (features change over time: PSI — Population Stability Index, KS test, distribution distances). Concept drift (relationship features→target changes). Prediction drift (output distribution changes). Performance metrics (if ground truth available with known delay — fraud: weeks to know false negative. CTR: hours to days). Data quality alerts
- **Retraining**: trigger on drift or schedule. Automated pipeline (trigger retrain → run validation tests → deploy if passes). Stale model risk: "model decays with time"

## MLOps Tools
- **MLflow**: open-source, tracking (experiment + parameters + metrics + artifacts), models (package + deploy), registry (manage model versions, stage transitions: Staging → Production → Archived). Model flavors: sklearn, pytorch, tensorflow, onnx, custom
- **Kubeflow**: ML workflows on Kubernetes (K8s-native). Pipelines (KFP: DAG of components, each component = container). Notebooks (Jupyter on K8s). Katib (hyperparameter tuning). KServe (model serving: inference with autoscaling + canary)
- **Weights & Biases**: experiment tracking + hyperparameter sweeps + model registry. Easy to set up (few lines of code). Integration with all major frameworks. Proprietary, SaaS (or self-hosted for enterprise). Good for + visualization comparisons
- **DVC**: data version control — Git for data (versions datasets + models). Store data in cloud (S3/GCS) + pointer in Git (small .dvc files). Pipeline (DAG of stages, each stage = command + inputs + outputs). Reproduce (run full pipeline, check if dependencies changed). Experiment tracking (git-based)

## Model Serving Infrastructure
- **Batch inference**: scheduled job (Airflow, Prefect, Dagster, scheduled container). Read data from source, run model, write predictions to DB/file. Good for: recommendations, credit scoring, fraud detection (pre-screening). Simpler than real-time
- **Real-time API**: REST/gRPC. Typical tech: FastAPI + Docker → K8s/SageMaker/ Sagemaker/ Seldon/ BentoML. Use BentoML (package model + pre/post-processing + serving, run as container). Use NVIDIA Triton Inference Server for GPU-optimized inference (multiple frameworks, dynamic batching). Use ONNX Runtime for cross-framework model format (convert PyTorch/TF/Sklearn to ONNX for optimized inference)
- **Serverless**: AWS Lambda (for lightweight models). Great for: low traffic, burstable, no server management. Limit: 15 min timeout, 1GB deployment package, cold starts (models loaded on each invocation = latency penalty). For low traffic: can be cost-effective
- **GPU inference**: GPU for deep learning inference (batch prediction with high throughput). High cost but low latency. GPU inference: Nvidia Triton, TensorRT (NVIDIA's optimization engine — quantize to FP16/INT8 for 2-4x speedup). Use for real-time + large models

## MLOps Culture
- **ML ≠ software**: data = dependency (different from code dependencies — data changes without changing code). Model behavior: non-deterministic (can't test with traditional assertions). Performance monitoring in production vs offline: offline metrics may not predict online behavior (because training-serving skew + concept drift). ML projects fail more often due to data + metric issues than model issues. The algorithm matters less than the data quality
- **Reproducibility**: code, data, environment, hyperparameters all versioned. Reproduce the trained model from the exact same state. Reproduce = run build pipeline again from snapshot versions. Data versioning: store data hash in experiment metadata, reference back to exact dataset snapshot. If you can't reproduce your model from scratch, you have a technical debt
- **Testing ML**: data tests (distribution, schema, nulls, ranges, class balance). Model tests (inference on known edge cases, invariance test, NA input test). Infrastructure tests (latency, throughput, memory). Integration test: end-to-end pipeline
- **Serving infrastructure**: A/B testing (randomize users, compare metrics). Feature store (avoid training-serving skew: compute transformations consistently between training + serving). Monitoring (drift detection = monitor feature + prediction distributions over time. Data quality checks (null rate, min/max/mean/median, schema violations, correlation shifts))
