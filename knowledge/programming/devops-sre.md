# DevOps & Site Reliability Engineering (SRE)

## DevOps Principles
- **CAMS**: Culture (collaboration, shared responsibility), Automation (infrastructure as code, CI/CD), Measurement (metrics, monitoring, data-driven decisions), Sharing (knowledge, blamelessness, transparency)
  - "DevOps is not a role, it's a culture" — but in practice many companies hire "DevOps engineers" 🤷
- **CALMS**: Culture, Automation, Lean (small batches, continuous improvement), Measurement, Sharing. CALMS is a more formal framework that emerged from the "Phoenix Project" / "State of DevOps" reports
- **The Three Ways**(Gene Kim, The Phoenix Project): 1. Flow (fast flow from dev to ops: small batches, automated, visual management). 2. Feedback (fast feedback from ops to dev: monitoring, alerting, blameless postmortems). 3. Continuous Learning & Experimentation (culture of improvement, safe experimentation, practice drills)
- **Continuous Delivery**: software always deployable. Small batches, automated testing, automated deployment pipeline. "Deploy on Monday." Deploy from trunk/main, not branching. Blue/green or canary to reduce risk

## SRE (Site Reliability Engineering)
- **Originated at Google** (Ben Treynor Sloss, 2003). SRE = "what happens when you ask a software engineer to design an operations team." Software engineering applied to ops
- **Error budget**: 100% - SLO = error budget. Spend error budget on risk (deploy quickly, new features) as long as SLO is met. If error budget is running low: slow down releases, focus on reliability
  - Example: 99.9% SLO (8.7h downtime/year) = 8.7h error budget. Used up 4h this month in incidents → 4.7h remaining. Decision: stop risky deployments for rest of month? The SLO is the internal target, not the SLA
- **Toil**: manual, repetitive, automatable, no enduring value work. SRE should spend <50% time on operational work (toil). If >50%, they're doing ops, not SRE. Fix: automate, eliminate, or hand off to dev team
- **SLO/SLI/SLA**: Service Level Indicator (measured — latency p99 < 500ms). Service Level Objective (target — 99.9% uptime). Service Level Agreement (contract — if SLO violated, pay penalty). SLI measurement: request latency, error rate (5xx), availability (uptime), throughput (# requests). SLO: internal target, tighter than SLA
  - "Happy SLO" metric: number of "good events" / total events. Target: 99.9% of events are good. Example: SLI = successful HTTP requests / total requests over 30 day rolling window. SLO = 99.9%. If error budget = 0.1% = ~0.1% of all outputs are lost to slowness/failures. When error budget is gone (SLO violated): don't deploy, stop features, maintain reliability

## Observability (Mature)
- **Pillars**: logs, metrics, traces — but observability = ability to ask NEW questions without shipping new code ("black box" vs "white box" monitoring). High cardinality data (user_id, request_id, session_id, feature_flag) — fast, flexible queries at scale. Move from "known unknowns" (dashboard predefined) to "unknown unknowns" (ad hoc queries to debug a rare issue)
- **Structured logging**: JSON format with event IDs, request IDs, resource IDs. Use correlation ID across services (e.g. x-request-id: 1234 — across API gateway, service A, service B, database). Log levels: FATAL, ERROR, WARN, INFO, DEBUG. Sample at high volume endpoints. Log retention: hot 7-30 days for interactive search. Cold longer for compliance queries
- **Metrics cardinality**: high cardinality = many label values per metric. Prometheus can handle millions per metric moderately OK, but high cardinality causes high memory usage. Use Exemplars: link metrics to traces for context. Rate, errors, duration (RED method). USE: utilization, saturation, errors. Rate: requests/second. Errors: failed requests/second. Duration: latency percentiles (p50, p95, p99, p999). Each metric has resource labels (host, service, region, deployment)
- **Tracing**: distributed tracing across services (OpenTelemetry, Jaeger, Zipkin). Trace contains spans for each service call. Each span = parent span (caller) + child span (service call) with timing + metadata. Trace waterfall: visualization of each service's latency, identify bottlenecks. Tail-based sampling: keep traces that have errors or high latency for deep debugging, drop fast success traces. Head-based sampling: sample X% of ALL traces (simpler, but misses rare errors). Intelligent sampling: keep ALL error traces + sample 1% successful

## Infrastructure as Code
- **Terraform**: declarative, state-driven, cloud-agnostic (AWS, GCP, Azure, etc.). HCL (HashiCorp Configuration Language). State file (terraform.tfstate): stores mapping between config + real resources — sensitive! Store in remote backend (S3 + DynamoDB lock). Workspaces: separate state for each environment (dev/staging/prod). Plan → Apply workflow. Modules for reusable components
  - Terraform best practices: use modules for reusable infrastructure (VPC, database, compute). Use remote state (S3/GCS/Azure Storage, locking via DynamoDB/Azure Storage blob lease). Don't use "count" for multiple environments — use separate workspaces or directories. Pin provider versions
- **Pulumi**: Terraform alternative — use real programming languages (TypeScript, Python, Go, C#). Same concepts (state, resources, providers) but code, not DSL. Better testing ability, reuse via actual functions/classes. More accessible to devs
- **Ansible**: configuration management + automation. Agentless (SSH), push-based. YAML playbooks (idempotent tasks). Roles (reusable playbook packages). Inventory (hosts + groups). Variables + templates (Jinja2). Not as good for infrastructure provisioning (Terraform better), great for server configuration (installing packages, config files, starting services). Pull vs push model: Ansible pushes. Alternatives: Puppet (pull), Chef (push), SaltStack (push/pull)
- **Docker + Docker Compose**: container run + networking. Dockerfile (build) + compose (run multi-service). Docker Compose for dev + small prod. Docker Swarm (built-in multi-node orchestration, simpler than K8s, less features). For production: K8s

## CI/CD Pipelines
- **GitHub Actions**: YAML, event-driven. Runners: GitHub-hosted (free 2000 min/month, ubuntu/macos/windows) or self-hosted. Matrix builds (multiple OS, language versions). Workflow: on push, on pull_request, schedule (cron), on manual dispatch. Jobs: run sequentially or in parallel. Steps: actions, scripts. Actions marketplace: community maintained, do one thing well. Actions cache ($ free for some? composite actions)
- **GitLab CI/CD**: .gitlab-ci.yml, pipelines, stages, jobs, GitLab Runner (auto-scaling, any executor: Docker, KVM, shell, Kubernetes, SSH). Merge request pipelines. Auto DevOps: detect language + build + test + deploy (limited but quick setup)
- **Jenkins**: older, most flexible, self-hosted, plugin-heavy (1,000+), Groovy pipeline DSL (declarative + scripted). Declarative: structured, simpler, better for most. Scripted: full programming (loop, try/catch), powerful but complex. Blue Ocean UI: modern interface. Slower startup, more maintenance
- **Pipeline stages**: checkout (pull code) → build (compile, package) → test (unit, integration, lint, security SAST, container scan) → deploy (to staging) → acceptance tests (E2E) → deploy (to prod) — gated by approval for prod. Some include "canary" stage (deploy to 5% of users, monitor X hours, roll forward to 100% or back). Promote to prod with manual approval

## SRE & Operations Maturity
- **Incident management**: declare (vs not — err on side of declaring, can always downgrade). Best practice: automatically create incident room, notify on-call. Communication to stakeholders (status page). Postmortem within 48h. Blameless analysis
- **Change management**: peer review required for ALL production changes. Change advisory board (CAB) for high-risk changes. Feature flags (deploy dark, enable for subset, ramp up, rollout). Rollback plan for every change. "Change freeze" during holiday season for many orgs
- **Capacity planning**: monitor resource utilization trends. Forecast growth based on user growth projections. Build 6-12 months ahead. Cost optimization: turn off non-prod resources nights/weekends. Use autoscaling. Right-size instances (use reserved instances for steady-state load, spot for variable)
- **Chaos engineering**: tooling (Chaos Monkey, Litmus, Chaos Mesh, Gremlin). Failure modes: instance failure, network latency, packet loss, AZ failure, DNS failure, expired certificate. Start: kill one non-critical service + observe. Progress accordingly
